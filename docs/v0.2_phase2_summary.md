## Phase 1 — HTTP/SSE Transport Interception

### Goal

> The full trifecta demo runs against an HTTP MCP server, not just STDIO.

**Achieved.** Agents connect via [Streamable HTTP 2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25/basic/transports/streamable-http). The same `InterceptedEvent` → engine path as STDIO. Backend MCP servers remain STDIO child processes (eBPF PID watching unchanged).

### What Was Built

| File / area | Purpose |
|---|---|
| `internal/proxy/dispatch.go` | Transport-agnostic `HandleAgentRequest` — initialize, tools/list, ping, tools/call routing, enforcement, sync wait for HTTP inspect-then-forward |
| `internal/proxy/http/transport.go` | `POST /mcp` handler: Origin validation, Accept header, session binding, JSON/SSE responses |
| `internal/proxy/http/response.go` | `WriteResponse` — blocked calls always JSON; success path may use SSE |
| `internal/proxy/http/headers.go` | MCP header validation (SEP-2243 baseline), auth/cookie redaction |
| `internal/proxy/http/client.go` | Demo/test Streamable HTTP client |
| `internal/config/config.go` | `transport.mode` (`stdio` \| `http`), `listen`, `endpoint`, `protocol_version`, `prefer_sse_responses` |
| `interlock-http.yaml`, `interlock-http-monitor.yaml` | HTTP transport example configs |
| `cmd/demo/http_transport.go` | HTTP demo passes (monitor / block / eBPF over HTTP) |

### Core design decision — inspect-then-forward

A blocking firewall cannot stream bytes to the agent before policy runs:

| Direction | Rule |
|---|---|
| Agent → Interlock | Full JSON-RPC body parsed before dispatch |
| Interlock → STDIO child | Hold-before-forward on `tools/call` (unchanged) |
| STDIO child → Interlock | Full result frame before `IngestResult` |
| Interlock → Agent (SSE) | Buffer complete JSON-RPC response before first SSE `data:` line |

Blocked `tools/call` → `Content-Type: application/json` with synthesized error immediately (no SSE).

**TLS posture (Phase 1):** bind `127.0.0.1` only — sit inside the trust boundary. TLS/MITM deferred.

**Deferred:** HTTP upstream backends, multi-session HTTP (→ Phase 2), GET `/mcp` listen streams, 2026-07-28 stateless protocol.

## Phase 2 — Multi-Session Concurrency and Attribution

### Goal

> Two poisoned sessions run **concurrently**, each correctly attributed, neither leaking state into the other.

**Achieved.** Each HTTP `initialize` spawns an isolated backend server pool. Trifecta state is keyed by `session_id` in the engine. eBPF events resolve to the correct session via `PIDRegistry` before `IngestSyscall`.

### What Was Built

| File | Lines (approx) | Purpose |
|---|---|---|
| `internal/proxy/session_manager.go` | 357 | `SessionManager` + `SessionRuntime`: per-session server pools, MCP session binding, idle sweeper, spawn/cleanup lifecycle |
| `internal/proxy/pid_registry.go` | 115 | Thread-safe `(pid, start_time)` → `{session_id, server_id}`; PID reuse safety via process liveness |
| `internal/proxy/proc_linux.go` | 39 | Read process start time from `/proc/<pid>/stat` |
| `internal/proxy/dispatch.go` | 237 | Refactored: `HandleAgentRequest(ctx, rt, frame)` — no global session pointer |
| `internal/proxy/proxy.go` | ~280 | STDIO single session via manager; HTTP via `StartHTTP` |
| `internal/ebpf/loader.go` | +6 | `RemovePID` for dynamic BPF map cleanup |
| `internal/ebpf/sensor.go` | +11 | `RemovePIDs`; watch/unwatch on session lifecycle |
| `internal/engine/session_store.go` | — | `Delete`; removed `FirstSessionID` |
| `internal/engine/engine.go` | — | `IngestSyscall` requires explicit `SessionID` — no session guessing |
| `internal/config/config.go` | — | `sessions.max_concurrent` (default 32), `sessions.idle_timeout` (default 30m) |
| `cmd/demo/http_concurrent.go` | 109 | Two parallel HTTP sessions, both blocked |
| `internal/proxy/http/concurrent_integration_test.go` | 174 | `TestConcurrentDualSession_VariantA_Block` |

### Why per-session server pools

Phase 1 HTTP created distinct MCP sessions but **shared one set of backend PIDs**. Under concurrency:

- Variant A could attribute proxy events correctly via `SetSession`, but handlers **raced** on a global `p.session`.
- Variant B could not attribute `connect()` to session N when all sessions shared the same exfil PID.

**Fix:** spawn tickets/messenger/exfil **per Interlock session** on HTTP `initialize`; register each child's PID in `PIDRegistry`; tear down on idle expiry or cleanup.

STDIO mode unchanged: one agent on stdin → one `SessionRuntime`.

### Architecture (after Phase 2)

```
                    ┌── HTTP client A ──POST /mcp──┐
                    │                             │
Agent(s) ───────────┤                             ├──► HTTP transport
                    │                             │         │
                    └── HTTP client B ──POST /mcp──┘         │
                                                             ▼
                                                    SessionManager
                                                     │         │
                              ┌──────────────────────┘         └──────────────────────┐
                              ▼                                                        ▼
                    SessionRuntime A                                          SessionRuntime B
                    (tickets/messenger/exfil PIDs)                             (separate PIDs)
                              │                                                        │
                              └──────────────► engine (per session_id) ◄───────────────┘
                                                    ▲
                                                    │ SessionID from PIDRegistry
                                              eBPF sensor (connect)
```

**Two planes, one engine — now session-aware on both paths.**

## Scope Decisions

**Agent-facing HTTP only; backends stay STDIO.** eBPF Variant B depends on watching child PIDs spawned by the proxy. Remote HTTP MCP backends would bypass the PID map — deferred.

**Inspect-then-forward over forward-then-inspect.** Correctness for a blocking firewall beats streaming latency. Documented in [architecture.md](architecture.md) §4.1.

**PID + start time, not PID alone.** OS PID reuse can attach a syscall to the wrong session if the map key is only PID. `/proc` start time disambiguates; liveness check breaks ties.

**No `FirstSessionID` fallback.** Silent session guessing under concurrency is a correctness bug. Unattributed eBPF events are audit-logged with full syscall detail — not guessed, not silently dropped.

**Session idle expiry default 30m.** Prevents orphaned server pools; configurable via `sessions.idle_timeout`.

**Per-session pools are a bounded DoS surface.** Each HTTP `initialize` spawns a full backend pool until idle expiry or `max_concurrent`. An adversary who can open sessions can hold up to `max_concurrent × servers` processes. Mitigate with auth in front of Interlock, lower limits, shorter idle timeout. Documented in README honest limitations.

## What's Next — Phases 3 & 4

### Phase 3 — Real Dataflow Taint

Closes the detection-credibility gap:

- Track tainted values through base64, hex, URL-encoding, reversal
- Make `TestCheckOverlap_EncodedExfil_KnownGap` **pass**
- Optional `sendto`/`write` payload capture → upgrade Variant B from `SUSPICIOUS` (0.60) to `EXFIL` (0.95)

**Done when:** a base64'd token in the sink is caught.

### Phase 4 — Performance, Benchmarks, and Persistent Evidence

The "is this operable" gate:

- Benchmark suite: p99 overhead per tool call, eBPF event throughput, degradation under load
- Replace JSONL-only evidence with embedded persistent store (SQLite first)
- Backpressure policy; instrument eBPF ring-buffer drop counts

**Done when:** published overhead number + evidence survives restart without unbounded growth.

### v0.2 exit state (full milestone)

When all four phases land: works on HTTP/SSE, handles concurrent sessions, catches encoded exfil, has published overhead numbers, persists evidence — **a usable tool** someone can point at a real agent. Tag **`v0.2.0`** when ready.
