# Detection and False-Positive Rate — Benign/Malicious Corpus

Generated by `internal/corpus` (also exercised in `internal/corpus/corpus_test.go`, CI-safe — no root/BTF/kind required). Regenerate with `make fp-corpus`.

## What these numbers are and are not

Every scenario drives `internal/engine.Engine` directly — `IngestResult`/`EvaluateRequest` for the proxy (Variant A), `IngestSyscall` for eBPF (Variant B), `IngestSyscallSensor` for the v0.3 Phase 1 sensor-only mode. There is no MCP proxy, no kernel, no network in this corpus; eBPF allowlist filtering happens in `internal/ebpf/sensor.go` before an event ever reaches the engine, so a corpus "non-allowlisted connect" is exactly what the engine would see in production.

Two outcome dimensions are scored per scenario, and they answer different questions:

- **TrippedAny** — any step returned a non-empty verdict (`SUSPICIOUS` or `EXFIL`). After ROADMAP §1, only `EXFIL` hard-blocks/kills; `SUSPICIOUS` emits evidence with `allowed_monitor` / `detected_only`. TrippedAny still counts verdict presence (including soft SUSPICIOUS).
- **TrippedExfil** — a step specifically reached verdict `EXFIL`, meaning `CheckOverlap`/`CheckOverlapPayload` proved a tainted value appears in the sink args/payload.

**Detection rate** below is EXFIL-tier (excludes documented `KnownGap` scenarios from the denominator — a catalogued gap is not a regression). **False-positive rate** is any-trip on benign (including soft SUSPICIOUS), with the EXFIL-tier false-positive rate reported separately since it should be, and currently is, zero.

## Discussion — does this reshape the logic?

**EXFIL-tier precision is high.** The false-positive rate at the EXFIL tier is 0.0% — value-overlap matching only fires when a tainted value's exact literal or one of its nine canonical encoded forms is actually present. No benign scenario in this corpus accidentally satisfies that bar, including ones deliberately constructed to look encoding-adjacent (base64 of unrelated data, masked previews, ticket-ID-only references).

**ROADMAP §1 remediated the sticky-leg tripwire.** `untrusted_content_present` no longer lights on sensitive-source results; `SUSPICIOUS` requires AllLit plus byte-level content-bind between untrusted excerpts and the sink; hard block/kill is reserved for `EXFIL`. Former ExpectTripByDesign sticky-leg scenarios are now true negatives. Remaining any-trip rows (if any) are intentional soft-SUSPICIOUS pins (`allowed_monitor` / `detected_only`), not hard-enforcement false positives.

**Leg decay** (`trifecta.leg_ttl` / `decay_after_calls`) dims sticky legs after idle time or N subsequent events so a poisoned session does not forever treat every sink as suspicious. Tainted values are retained so a late sink that still carries a secret can still reach EXFIL. The corpus exercises both dimensions (`benign_proxy_a_leg_decay_then_sink`, `benign_proxy_a_high_throughput_50_then_sink`, TTL past-boundary `benign_proxy_a_leg_ttl_decay_then_sink` at 31m, and still-lit under-boundary `benign_proxy_a_leg_ttl_still_lit_under_boundary` at 29m59s via `StepAdvanceTime`).

**Extraction boundary.** `extractResultText` prefers MCP `content[].text`, then walks other JSON string leaves (bounded). Benign nested-metadata scenarios keep an unrelated sink so registered metadata taint does not EXFIL without overlap; the paired malicious scenario (`malicious_proxy_a_secret_outside_content_text`) proves buried secrets are tainted. Soft-SUSPICIOUS any-trip pins (natural product quotes, multi-server B→C binds) are intentional `allowed_monitor` noise for fetch-heavy agents — tune `content_bind_min_len` rather than hard-blocking.
